# BOJ 3085 - 사탕 게임

## 문제 요약

N*N 보드에서 인접한 두 사탕을 교환한 뒤, 같은 색으로 이루어진 가장 긴 연속 부분(행 또는 열)의 길이를 구하는 문제.

## solution.js 리뷰

### 정확성

로직은 정확하다. 가로 인접 스왑, 세로 인접 스왑 각각에 대해 전체 보드에서 최대 연속 길이를 계산한다.

### 시간 복잡도: O(N^4)

- 스왑 후보: O(N^2)
- 각 스왑마다 `calculateMax`가 전체 보드 순회: O(N^2)
- N=50일 때 약 6,250,000번 연산으로 통과는 하지만 비효율적

### 코드 스타일 지적 사항

1. **원본 배열 mutation**: swap-restore 패턴은 복원 실수 시 디버깅이 어렵다. 실제로 세로 스왑 인덱스를 잘못 써서 버그가 발생했었다.
2. **calculateMax 내부 로직 중복**: 행 순회와 열 순회가 인덱스 순서만 다르고 동일한 로직이다.
3. **불필요한 `j === 0` 분기**: `currentValue`를 루프 전에 초기화하면 제거 가능하다.

## solution2.js - 전문가 풀이

### 핵심 아이디어

스왑은 최대 2개 행, 2개 열에만 영향을 준다. 전체 보드를 매번 스캔하지 않고 영향받는 행/열만 검사하면 O(N^3)으로 최적화할 수 있다.

### 영향 범위 분석

| 스왑 종류 | 검사 대상 |
|-----------|-----------|
| 가로 (i,j)↔(i,j+1) | i행, j열, j+1열 |
| 세로 (i,j)↔(i+1,j) | i행, i+1행, j열 |

### 주의사항: baseline 최대값

영향 범위만 검사하는 최적화를 할 때, 같은 색이라 스왑을 건너뛰는 쌍에서 이미 긴 연속이 존재할 수 있다. 따라서 스왑 전에 전체 행/열의 기본 최대값(baseline)을 먼저 구해야 한다.

### 설계 포인트

- `maxInRow(r)`: 특정 행의 최대 연속 길이
- `maxInCol(c)`: 특정 열의 최대 연속 길이
- `swap(r1, c1, r2, c2)`: 동일 함수를 두 번 호출하면 swap + restore가 된다
- 스왑 루프 전에 baseline 최대값을 먼저 계산
- 루프 변수 `i`=행, `j`=열로 일관되게 사용

### 비교 표

| | solution.js | solution2.js |
|---|---|---|
| 시간 복잡도 | O(N^4) | O(N^3) |
| 스왑 후 검사 범위 | 전체 N*N 보드 | 영향받는 2~3줄 |
| 코드 중복 | 행/열 로직 각각 작성 | `maxInRow`/`maxInCol` 분리 |
| mutation 복원 | 수동 3줄 복원 | `swap` 함수 동일 호출 2번 |
| 루프 변수 혼동 위험 | i/j 역할이 블록마다 다름 | i=행, j=열로 일관 |

## 핵심 교훈

"영향 범위만 검사"하는 사고방식은 브루트포스 최적화에서 자주 쓰이는 패턴이다. 전체를 다시 계산하기 전에 "이 변경이 실제로 영향을 주는 범위가 어디인가?"를 먼저 생각하자.
